// Protocol Comparison

WireGuard or OpenVPN?
Here's the honest answer.

Choose WireGuard unless you have a specific reason not to. If your organisation has an existing OpenVPN deployment or a compliance requirement specifying the protocol, go OpenVPN.

Recommended
WireGuard
SpeedVery fast — minimal overhead, built into Linux kernel
LatencyExcellent — UDP-only, no handshake overhead per packet
Battery lifeBest — efficient on mobile, low CPU on reconnect
ReconnectInstant — roams between networks seamlessly
Config simplicityVery simple — one config file per device, easy to deploy
Audit historyNewer — less legacy audit history than OpenVPN
Firewall bypassUDP only — may be blocked on strict corporate firewalls
CompatibilityAll platforms — iOS, Android, Windows, macOS, Linux
OpenVPN
SpeedGood — slightly slower than WireGuard due to userspace
LatencyGood — slightly higher overhead per connection
Battery lifeOK — higher CPU usage on reconnect vs WireGuard
ReconnectGood — reliable but slower handshake
Config simplicityMore complex — certificate management, more config options
Audit historyExtensive — 20+ years of security audits
Firewall bypassTCP mode available — can run on port 443 to bypass firewalls
CompatibilityAll platforms — very wide existing client support
// Our recommendation: If you're starting fresh — use WireGuard. It's faster, simpler to manage, and performs better on mobile networks like Safaricom and Airtel. Choose OpenVPN if you're integrating with an existing deployment or have a policy requirement. Not sure? Ask us — we'll give you a straight answer.

Automated patches · 48hr notice

Split Tunnelling
Configure which traffic routes through the VPN and which goes direct. Route only internal traffic (e.g. 10.0.0.0/8) through the tunnel, keeping general internet traffic on the local connection.
Configurable routing · per-client
[kill]
Kill Switch Config
We include kill-switch configuration in client configs — if the VPN drops, the device blocks all internet traffic until reconnected. Critical for staff handling sensitive data on public Wi-Fi.
Optional · included in config

The VPN server sits on our KIXP-peered network. Kenyan users on Safaricom, Airtel, or Telkom get 2–5ms latency to the VPN endpoint — the same network as our VM infrastructure.
~2ms from Safaricom · KIXP direct

The VPN server and all logs are in Kenya. Traffic routed through our endpoint stays in-country. For NGOs or regulated organisations with Kenya data-residency requirements, we provide written confirmation.

// FAQ

VPN questions.

Can't find your answer? Email us directly.

What's the difference between a managed VPN and a self-hosted one?+

If you deploy WireGuard on a Pylon VM yourself, you own everything — you configure it, maintain it, patch it, and debug it when something breaks at 11pm. That's fine if you have someone who knows Linux.

With a managed VPN, we provision the server, handle all server-side configuration, monitor it 24/7, apply OS and WireGuard updates, manage key rotation, and generate client configs. You just distribute the config files to your team. No sysadmin required on your side.

How does my team connect from a mobile device?+

We send each user a WireGuard config file (or a QR code for easy mobile import). On iOS or Android, they install the free WireGuard app, scan the QR code, and tap Connect. The entire setup takes under three minutes per device — no IT knowledge needed.

WireGuard handles network switching (Wi-Fi to 4G) transparently, so it's especially good for staff who move between Safaricom data and office Wi-Fi throughout the day.

Can I restrict which internal servers each user can access?+

Yes. WireGuard's AllowedIPs directive lets us specify per-peer which subnets are accessible through the tunnel. We can configure different access levels for different users — for example, developers get access to all internal servers, while field staff only see the CRM server at 10.0.0.50.

More complex access policies (role-based, time-limited) can be implemented on the Enterprise plan with firewall rules on the VPN server.

What happens if a staff member leaves or loses their laptop?+

Email or WhatsApp us with the name or peer ID to revoke. We remove their public key from the server configuration within the hour — they can no longer connect, even with the original config file. Their config file becomes useless instantly.

For high-turnover teams or situations where you need self-service revocation, the Enterprise plan with admin dashboard access is more appropriate.

Can the VPN handle staff connecting from Uganda, Tanzania, or Rwanda?+

Yes — WireGuard works from anywhere with an internet connection. Staff in Kampala, Dar es Salaam, or Kigali connect to the Nairobi VPN server over the internet. Latency will be higher than for Nairobi users (typically 30–60ms from East African capitals vs 2–5ms from within Kenya), but entirely functional for accessing internal systems, databases, and shared resources.

Is the VPN traffic logged?+

We maintain server-side connection logs (who connected, when, for how long, from which IP) for security and troubleshooting purposes, retained for 30 days by default. We don't log the content of VPN traffic — only connection metadata.

These logs are stored in Kenya and subject to KDPA. We don't share them with third parties without a valid legal order. For organisations that require zero connection logging, the Enterprise plan can be configured accordingly — contact us to discuss.

Can the VPN server see my team's internet traffic?+

It depends on how split tunnelling is configured. With full-tunnel routing (all traffic through VPN), yes — we could see all internet traffic from connected devices. With split tunnelling (only internal subnets routed through the VPN), only traffic to your internal network passes through our server.

We configure split tunnelling by default unless you specifically request full tunnel. We recommend split tunnelling for most use cases — it's faster, uses less bandwidth, and means we only see internal traffic.
